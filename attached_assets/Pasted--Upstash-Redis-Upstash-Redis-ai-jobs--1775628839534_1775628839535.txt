# 진로 흥미 분석 — Upstash Redis 큐 적용 지시서

## 배경

현재 "진로 진단" 메뉴는 Upstash Redis 큐 + ai_jobs 테이블 기반의 비동기 처리 패턴이 적용되어 있어 다수 사용자가 동시에 요청해도 안정적으로 처리됩니다.

"진로 흥미 분석"(`POST /api/aptitude/analyze`)도 LLM(Gemini)을 호출하므로 동일한 큐 패턴을 적용합니다. 단, 진로 흥미 분석의 LLM 부하는 진로 진단보다 가벼우므로(추천 사유 + 요약만 생성, 예상 3~8초) 설정을 그에 맞게 조정합니다.

---

## 작업 내용

### 1. 기존 진로 진단의 큐 패턴을 진로 흥미 분석에 복제 적용

진로 진단에서 이미 구현된 아래 흐름을 그대로 가져옵니다:

```
[프론트엔드]                    [서버]                         [워커]
제출 클릭 → POST /api/aptitude/analyze
                            → ai_jobs에 pending INSERT
                            → Redis 큐에 jobId 발행
                            → jobId 반환
                            
폴링 시작 → GET /api/aptitude/job/:jobId
                                                    큐에서 jobId 소비
                                                    → ai_jobs를 processing으로
                                                    → LLM 호출 (3~8초)
                                                    → DB 트랜잭션 {
                                                        aptitudeAnalyses INSERT
                                                        ai_jobs를 completed로
                                                      }
                                                    → 완료
폴링 응답 ← completed + 결과 데이터
결과 화면 표시
```

### 2. 핵심 구현 사항 (3가지)

#### 2-1. DB 트랜잭션으로 묶기

워커에서 LLM 응답을 받은 후, `ai_jobs` 상태 업데이트와 `aptitudeAnalyses` 삽입을 **반드시 하나의 트랜잭션**으로 처리합니다.

```typescript
// 워커 내부
await db.transaction(async (tx) => {
  // 1. 분석 결과 저장
  await tx.insert(aptitudeAnalyses).values({
    userId,
    interestScores: result.interestScores,
    abilityScores: result.abilityScores,
    recommendedJobs: result.recommendedJobs,
    recommendedMajors: result.recommendedMajors,
    summary: result.summary,
    // ...기타 필드
  });
  
  // 2. ai_jobs 상태를 completed로 업데이트
  await tx.update(aiJobs)
    .set({ status: 'completed', completedAt: new Date() })
    .where(eq(aiJobs.id, jobId));
});
```

이렇게 하면 "ai_jobs는 completed인데 분석 결과가 없는" 불일치 상태가 방지됩니다.

#### 2-2. 중복 제출 방지 가드

사용자가 연속으로 제출하거나 새로고침으로 재제출하는 경우, 큐에 중복 작업이 쌓이지 않도록 합니다.

```typescript
// POST /api/aptitude/analyze 라우트 시작부
const existingJob = await db.select()
  .from(aiJobs)
  .where(
    and(
      eq(aiJobs.userId, userId),
      eq(aiJobs.type, 'aptitude_analysis'),  // 진로 흥미 분석 전용 타입
      inArray(aiJobs.status, ['pending', 'processing'])
    )
  )
  .limit(1);

if (existingJob.length > 0) {
  // 이미 진행 중인 작업이 있으면 해당 jobId 반환 (새 작업 생성하지 않음)
  return res.json({ jobId: existingJob[0].id, status: existingJob[0].status });
}
```

#### 2-3. 프론트엔드 폴링 전환

현재 진로 흥미 분석은 `POST` 후 바로 응답을 기다리는 동기 방식입니다. 이것을 jobId 기반 폴링으로 변경합니다.

```typescript
// Aptitude.tsx — 제출 핸들러
const handleSubmit = async () => {
  setAnalyzing(true);
  
  // 1. 큐에 작업 등록, jobId 받기
  const { jobId } = await api.post('/api/aptitude/analyze', { answers, scores });
  
  // 2. 폴링으로 완료 대기
  const result = await pollJobResult(jobId);
  
  // 3. 결과 표시
  setAnalysisResult(result);
  setAnalyzing(false);
};

// 폴링 함수
async function pollJobResult(jobId: string, maxAttempts = 30) {
  for (let i = 0; i < maxAttempts; i++) {
    const { status, data } = await api.get(`/api/aptitude/job/${jobId}`);
    
    if (status === 'completed') return data;
    if (status === 'failed') throw new Error('분석 처리 중 오류가 발생했습니다');
    
    // 2초 간격 폴링 (3~8초 소요 예상이므로 2~4회 폴링으로 완료)
    await new Promise(r => setTimeout(r, 2000));
  }
  throw new Error('분석 시간이 초과되었습니다');
}
```

### 3. 설정값

| 설정 | 진로 진단 (참고) | 진로 흥미 분석 (신규) |
|------|----------------|-------------------|
| 큐 이름 | 기존 값 유지 | `aptitude-analysis` (별도 큐 또는 기존 큐에 type 구분) |
| ai_jobs.type | 기존 타입 | `aptitude_analysis` |
| 워커 timeout | 기존 값 | **60초** (LLM 응답 3~8초 + 여유) |
| 폴링 간격 | 기존 값 | **2초** |
| 폴링 최대 횟수 | 기존 값 | **30회** (최대 60초 대기) |
| 동시 실행 제한 | 기존 값 유지 | 기존과 동일하게 적용 |

### 4. 프론트엔드 UX

LLM 응답이 3~8초로 빠르므로, 복잡한 프로그레스 바 없이 단순한 "처리 중" 상태로 충분합니다.

```
분석 중인 상태:
┌─────────────────────────────┐
│                             │
│     🔄 분석 중입니다...       │
│     잠시만 기다려 주세요       │
│                             │
└─────────────────────────────┘
```

현재 이미 `analyzing` 상태에서 로딩 UI를 표시하고 있다면 그대로 유지하면 됩니다. 내부적으로만 동기 호출 → 폴링으로 바뀌는 것이므로 사용자가 느끼는 차이는 없습니다.

### 5. 에러 처리

```typescript
// 워커에서 LLM 호출 실패 시
try {
  const llmResult = await callGemini(prompt);
  // ... 트랜잭션으로 저장
} catch (error) {
  await db.update(aiJobs)
    .set({ 
      status: 'failed', 
      error: error.message,
      failedAt: new Date() 
    })
    .where(eq(aiJobs.id, jobId));
}
```

프론트엔드에서 `status === 'failed'`를 받으면 "분석 중 오류가 발생했습니다. 다시 시도해 주세요." 메시지를 표시합니다.

### 6. 이중 INSERT 방지

워커가 stale job을 재시도할 경우 동일 사용자에게 분석 결과가 2개 생성될 수 있습니다. 이를 방지하려면:

```typescript
// 워커 시작 시 이미 처리된 잡인지 확인
const job = await db.select().from(aiJobs).where(eq(aiJobs.id, jobId)).limit(1);

if (!job.length || job[0].status === 'completed') {
  // 이미 완료된 잡이면 스킵
  return;
}
```

## 수정 대상 파일

- `server/routes.ts` — `POST /api/aptitude/analyze` 라우트를 큐 발행 방식으로 변경, `GET /api/aptitude/job/:jobId` 폴링 엔드포인트 추가
- `server/worker.ts` (또는 기존 워커 파일) — 진로 흥미 분석 작업 처리 로직 추가
- `client/src/pages/Aptitude.tsx` — 동기 호출을 폴링 방식으로 변경
- `shared/schema.ts` — ai_jobs.type에 `aptitude_analysis` 값 추가 (필요 시)

## 주의사항

1. **기존 진로 진단 큐 로직을 건드리지 마세요** — 새로운 타입(`aptitude_analysis`)을 추가하는 것이지, 기존 로직을 수정하는 것이 아닙니다
2. **크레딧/환불 로직은 불필요** — 진로 흥미 분석은 무료이므로 진로 진단에 있는 결제/환불 관련 코드는 복제하지 않습니다
3. **기존 분석 결과 조회 API(`GET /api/aptitude/latest`)는 유지** — 대시보드에서 최근 결과를 표시하는 데 사용하므로 그대로 둡니다
