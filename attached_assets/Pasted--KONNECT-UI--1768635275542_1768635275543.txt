역할: 너는 KONNECT 웹앱의 프론트엔드(필요 시 백엔드 타입/스키마)까지 정리해서 수정하는 시니어 엔지니어다.
목표: 첨부된 ‘내 프로필’ 화면을 아래 요구사항대로 UI/필드/메뉴 구조를 개편하고, 기존 데이터 호환을 깨지 않도록 반영한다.

0) 먼저 프로젝트 구조 파악
- 아래 순서로 저장소를 스캔해서 “프로필 화면”과 “좌측 사이드바 메뉴”가 어디서 렌더링되는지 찾아라.
  - pages/, app/, src/pages/, src/app/ 중 라우팅 위치 확인
  - components/Sidebar, components/Nav, components/Profile, pages/profile, app/profile 등 키워드로 검색
  - “내 프로필”, “커리어 분석”, “목표 관리”, “자기소개서”, “직업 탐색” 문자열로 검색해서 사이드바/프로필 페이지 파일을 특정
- 프로필 폼이 React Hook Form / Formik / 자체 state 중 무엇인지 확인하고, validation(zod/yup) 및 API 호출 레이어도 함께 찾아라.

1) 프로필 유형 명칭 변경 및 좌측 메뉴 목록화
요구사항
- 프로필 유형에서 “일반(직장인)”을 “구직자”로 변경한다.
- 좌측 메뉴에서 “내 프로필” 옆(또는 하위)에 목록 형태로 다음 항목을 추가한다:
  - 구직자, 대학생, 고등학생, 중학생, 초등학생
동작 정의
- 좌측 메뉴는 “내 프로필”이 상위 메뉴이고, 클릭 시 하위 목록(서브메뉴)이 펼쳐지는 구조로 구현한다(아코디언/드롭다운).
- 하위 항목 클릭 시 프로필 유형이 해당 값으로 설정되며, URL 파라미터 또는 상태로 유지되게 한다.
  - 예: /profile?type=jobseeker, /profile?type=college … 같은 방식(프로젝트 라우팅 스타일에 맞춰 결정)
- 기존에 저장된 타입 값이 “general/worker” 류로 존재한다면:
  - 표시 라벨만 바꾸는지(데이터 값은 유지), 또는 데이터 값도 jobseeker로 마이그레이션할지 선택해야 한다.
  - 백엔드/DB에 의존성이 있으면 “표시 라벨만 변경 + 내부 값은 기존 유지”를 우선으로 하고, 별도 마이그레이션이 가능한 구조면 매핑 로직을 추가한다(깨짐 방지).
  - 어떤 방식을 택했는지 코드 주석과 커밋 메시지에 명확히 남겨라.

2) 우측 상단 프로필 기본정보를 가로 배치로 개편
요구사항
- 우측 화면 상단에 다음 필드를 가로로 나열해서 넣는다:
  - 이름, 메일주소, 소속국가, 생일/생년월일, 한줄소개
UI 구현 가이드(반응형)
- 데스크톱: 1행 가로 배치(예: CSS grid 5열 또는 6열, 한줄소개는 더 넓게 2칸 차지 가능)
- 태블릿/모바일: 2~3행으로 자연스럽게 줄바꿈
- 기존의 아바타 카드(원형 이니셜/프로필 이미지)와 이메일/지역/성별 등은 상단 가로 필드 레이아웃에 맞춰 정리한다.
  - 성별 필드는 요구사항에 없으므로 기본은 숨기되, 기존 데이터가 있으면 별도 “추가 정보” 섹션으로 이동시키는 방향으로 정리(삭제는 아님).
- 한줄소개는 input 또는 textarea(1~2줄 자동 확장)로 상단에 배치한다.

3) 현재 상태 및 이직/구직 의도 카드 필드 수정
요구사항
- “이전(또는 현재) 직무 만족도” 필드는 삭제한다(UI에서 제거, 저장 로직은 데이터 호환 위해 즉시 삭제하지 말고 사용하지 않게 처리).
- “새로운 일을 찾는 주된 이유” 라벨을 “구직사유”로 변경한다.
추가 정리
- 카드 내부 필드들을 보기 좋게 재배치한다(간격/정렬/그룹핑).
- “현재 구직 상태” 드롭다운은 유지하되, 라벨/옵션 문구가 어색하면 한국어 UX 관점에서 자연스럽게 다듬는다(의미 변경 금지).

4) 보유역량 및 스킬을 더 풍성하게, 경력은 별도 섹션으로 분리
요구사항
- “보유 역량 및 스킬” 선택지를 더 풍성하게 구성한다.
  - 최소: 카테고리(예: 기획/PM, 데이터/분석, 개발, 디자인, 마케팅/세일즈, 운영/CS, 문서/리서치, 언어/커뮤니케이션 등)
  - 다중 선택 가능, 검색 가능(스킬이 많아져도 찾기 쉽게)
  - 선택된 스킬은 태그(칩) 형태로 표시하고, 클릭으로 제거 가능
  - 필요하면 “직접 입력(커스텀 스킬 추가)”도 지원
- “경력”은 “보유 역량 및 스킬” 카드 안에 두지 말고, 별도 섹션(별도 카드 또는 별도 탭/페이지)으로 분리한다.
  - 예: 프로필 페이지 내 ‘경력’ 카드(추가/수정 리스트), 또는 /profile/experience 같은 하위 라우트
  - 기존의 “경력 추가하기” UI는 그대로 쓰되 위치만 분리한다.
데이터 처리
- 스킬 데이터가 문자열 배열이면 그대로 확장하고, 카테고리 구조가 필요하면 내부적으로 {category, items[]}를 UI용 상수로 두고 저장은 문자열 배열로 유지(호환 우선).
- 경력 데이터 구조가 기존에 있으면 그대로 유지하고 UI만 분리한다.

5) 폼/저장/유효성 검사 정리
- 상단 가로 필드(이름/메일/국가/생년월일/한줄소개)가 기존 폼 상태와 저장 API에 정확히 연결되게 한다.
- 저장 버튼(우측 상단 “저장하기”) 동작을 유지한다.
- 필드 삭제(직무만족도)는 UI에서 제거하되, 백엔드 스키마에 필드가 남아 있어도 에러가 나지 않게 한다.
- 입력 컴포넌트 간 간격, 레이블 정렬, 카드 여백을 통일한다.

6) 완료 기준(Acceptance Criteria)
- 좌측 메뉴에 “내 프로필” 하위로 구직자/대학생/고등학생/중학생/초등학생 목록이 표시되고, 선택 시 해당 유형이 반영된다.
- 프로필 유형 “일반(직장인)” 표기는 화면에서 사라지고 “구직자”로 대체된다.
- 우측 상단에 이름/메일/소속국가/생년월일/한줄소개가 가로 배치로 보이며 반응형에서 깨지지 않는다.
- “이전(또는 현재) 직무 만족도”는 UI에서 제거된다.
- “새로운 일을 찾는 주된 이유”는 “구직사유”로 변경된다.
- 보유역량/스킬은 카테고리+검색+태그 형태로 선택 가능하고, 선택지가 확장되어도 사용성이 유지된다.
- 경력은 별도 섹션으로 분리되어 보인다.
- 저장하기 버튼으로 저장 시 콘솔 에러/네트워크 에러 없이 동작한다(프로젝트의 기존 저장 방식 준수).

7) 실행 명령(프로젝트에 맞춰 자동 선택)
- 저장소에서 다음을 확인해서 패키지 매니저를 선택해 실행하라:
  - pnpm-lock.yaml 있으면: pnpm i && pnpm dev
  - yarn.lock 있으면: yarn && yarn dev
  - package-lock.json 있으면: npm i && npm run dev
- 린트/타입체크 스크립트가 있으면 함께 실행:
  - npm run lint / npm run typecheck / npm test 등 package.json의 scripts를 확인 후 수행

8) 변경 후 결과 보고
- 수정한 핵심 파일 목록과 변경 요약을 간단히 정리해서 출력하라.
- 화면 스크린샷을 찍을 수 있으면(before/after) 비교를 위해 첨부하라(가능한 경우).

이제 위 요구사항대로 실제 코드를 수정해라. 먼저 파일 위치를 찾아내고, 그 다음 단계별로 PR 단위로 깔끔하게 반영해라.
