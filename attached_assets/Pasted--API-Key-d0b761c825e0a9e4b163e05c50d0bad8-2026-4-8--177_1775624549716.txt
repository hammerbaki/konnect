### 1단계: 직업 데이터 전수 재수집 (최우선)

#### 1-1. 목록 전체 수집

`jobs.json` API를 pageIndex=1부터 56까지 순회하여 전체 552개 직업의 `job_cd`, `job_nm`, `wage`(텍스트), `job_category`를 수집합니다.

```typescript
// 의사 코드
const API_KEY = 'd0b761c825e0a9e4b163e05c50d0bad8';
const allJobs = [];

for (let page = 1; page <= 56; page++) {
  const url = `https://www.career.go.kr/cnet/front/openapi/jobs.json?apiKey=${API_KEY}&pageIndex=${page}&pageSize=10`;
  const res = await fetch(url);
  const data = await res.json();
  allJobs.push(...data.jobs);
  // Rate limiting: 200ms 대기
  await new Promise(r => setTimeout(r, 200));
}
console.log(`총 ${allJobs.length}개 직업 수집`); // 552 기대
```

#### 1-2. 상세 데이터 수집 (정밀 급여 + 관련 학과)

각 직업의 `job_cd`로 `job.json` 상세 API를 호출하여 정밀 데이터를 수집합니다.

```typescript

  const departList = detail.departList || []; // 관련 학과 [{depart_id, depart_name}]
  
  // DB 업데이트
  // salary: preciseWage에서 콤마 제거 후 숫자로 변환 (만원 → 원 변환 필요 시 *10000)
  // related_majors: departList에서 학과명 배열 추출
  // description: baseInfo.summary
  // 등등
  
  await new Promise(r => setTimeout(r, 200)); // Rate limiting

```

**급여 파싱 규칙:**
- `baseInfo.wage`가 `"3,833"` 형태 → 콤마 제거 후 `3833` (만원 단위 연봉)
- 일부 직업은 wage가 없거나 빈 문자열일 수 있음 → null 처리
- 기존 `cached_jobs.salary`는 **원(원) 단위** 저장 → `3833만원 = 38330000` 변환 여부는 기존 컬럼 형식에 맞출 것
- ⚠️ 목록 API의 `wage`("3천만원↓")는 대략적 범위이므로 상세 API의 `baseInfo.wage`를 우선 사용

**관련 학과 연결:**
- `departList[].depart_id`는 학과 API의 `majorSeq`와 동일한 값
- `departList[].depart_name`은 학과명
- 이 데이터로 `cached_jobs.related_majors` 필드를 정확하게 업데이트

## 주의사항

1. **Rate Limiting**: API 호출 간 최소 200ms 간격 유지. 552개 직업 상세 조회 시 약 2분 소요.
2. **에러 처리**: 일부 job_cd로 상세 조회 시 데이터가 없을 수 있음 → try/catch로 스킵 후 로그
3. **기존 데이터 보존**: `cached_jobs`의 `job_category` 필드는 이전에 수동으로 56건 수정했으므로, API에서 가져온 `job_category`가 있으면 이것으로 덮어쓰기 (API 값이 정확함)
4. **`normalizeMajorName()` 함수**: 이미 수정 완료된 상태 (`전공` → `과/부` 순서로 제거). 학과명 매칭 시 이 함수 사용.
5. **salary 단위 통일**: 기존 `cached_jobs.salary`가 원(원) 단위인지 만원 단위인지 확인 후, API 값(`만원`)과 맞출 것. 프론트엔드 표시 코드도 같이 수정.
6. **페이지네이션은 정상 작동합니다**: 이전에 "모든 페이지가 같은 데이터를 반환한다"는 보고가 있었지만, 직접 테스트 결과 pageIndex를 변경하면 다른 데이터가 정상 반환됩니다. 코드의 파라미터 전달 로직을 확인하세요.
